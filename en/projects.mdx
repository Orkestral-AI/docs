A project is the home for one codebase inside a workspace. It points at a real repository or folder, and it becomes the context your AI team works in. When you make a project active, the orchestrator and its specialists read that code, plan against it, and run the work as issues and epics.

Think of a project as the answer to "which code are we working on right now?". You can keep many projects in a workspace and switch between them as you move from one product or service to another.

<Info>
Projects live inside a **workspace**. If you have not set up a workspace yet, start with [Workspaces](/en/workspaces) and then come back here.
</Info>

## What a project is

Each project records a small, focused set of facts about one codebase:

<CardGroup cols={2}>
  <Card title="Name" icon="tag">
    A short label you recognize, like `billing-api` or `marketing-site`.
  </Card>
  <Card title="Type" icon="laptop">
    **Local** (a folder on your machine) or **GitHub** (a remote repository URL).
  </Card>
  <Card title="Folder or repo URL" icon="folder-open">
    The path on disk for a local project, or the Git remote for a GitHub project.
  </Card>
  <Card title="Description" icon="align-left">
    Optional context that helps your AI team understand the goal of the codebase.
  </Card>
</CardGroup>

A project is intentionally lightweight. It does not store your code (that stays in the repository), it stores the pointer to your code plus the metadata your agents need to reason about it.

## How a project groups your work

Once a project is active, everything you create flows into it. The orchestrator plans against the active project's code, and the work it produces is tracked as issues and epics tied to that project.

<CardGroup cols={3}>
  <Card title="Issues" icon="circle-dot">
    A single unit of work: a bug, a small feature, a change. See [Issues](/en/issues).
  </Card>
  <Card title="Epics" icon="layer-group">
    A larger goal split into several issues that ship together. See [Epics](/en/epics).
  </Card>
  <Card title="Chat" icon="comments">
    Conversations with your agents run against the active project's context.
  </Card>
</CardGroup>

This grouping is what keeps a busy workspace readable. When you switch projects, you switch the lens: the code being read, the plan being made, and the work being tracked all move with the active project.

<Note>
Only one project is active at a time. The active project is shown on its card as **Active project**, and it drives what the orchestrator and specialists see.
</Note>

## Create a project

<Steps>
  <Step title="Open the Projects page">
    From your active workspace, open **Projects**. You see a grid of existing projects, or an empty state if this is your first one.
  </Step>
  <Step title="Start a new project">
    Click **New project** in the top right. The empty state also has a **Create your first project** button that does the same thing.
  </Step>
  <Step title="Choose the type">
    Pick **Local** to point at a folder on your machine, or **GitHub** to point at a remote repository.

    <Tabs>
      <Tab title="Local">
        Use this for code that already lives on your computer. You will pick a folder in the next step.
      </Tab>
      <Tab title="GitHub">
        Use this for a repository hosted on GitHub. You will paste its repository URL instead of a folder.
      </Tab>
    </Tabs>
  </Step>
  <Step title="Name the project">
    Type a clear name. If you pick a folder first and leave the name empty, Orkestral fills the name from the last segment of the folder path automatically.
  </Step>
  <Step title="Point at the code">
    <Tabs>
      <Tab title="Local">
        Click **Choose** to open the native folder picker, or paste the path directly into the field. The path is shown in monospace so you can confirm it is correct.
      </Tab>
      <Tab title="GitHub">
        Paste the repository URL (for example a `https://` or `git@` remote) into the repository URL field.
      </Tab>
    </Tabs>
  </Step>
  <Step title="Add a description (optional)">
    Write a sentence or two about what the codebase does. Your AI team uses this as context, so a good description pays off in better plans.
  </Step>
  <Step title="Create it">
    Click the create button. While it saves you see a spinner and **Creating**. When it finishes, the dialog closes and the new project appears in the grid.
  </Step>
</Steps>

<Tip>
The name field is required. The create button stays disabled until you type a name, so if it looks stuck, check that the name is not empty.
</Tip>

## Make a project active

A project only drives your AI team once it is the active project.

<Steps>
  <Step title="Find the project card">
    On the **Projects** page, locate the card for the project you want to work in.
  </Step>
  <Step title="Make it active">
    Click **Make active** at the bottom of the card. The card highlights and the label changes to **Active project**.
  </Step>
  <Step title="Start working">
    Create issues, plan an epic, or open a chat. All of it now runs against this project's code.
  </Step>
</Steps>

<Warning>
Switching the active project changes the context for new work. Issues and epics you already created stay tied to the project they were made under, so switching does not move existing work.
</Warning>

## Read the card at a glance

Each project card surfaces the key facts without opening anything:

<AccordionGroup>
  <Accordion title="Name and folder" icon="folder-tree">
    The top of the card shows the project name. For local projects, the folder path appears just below it in monospace so you can confirm which checkout it points at.
  </Accordion>
  <Accordion title="Description" icon="align-left">
    If you added a description, it shows under the name, trimmed to a couple of lines. This is the quickest reminder of what the codebase is for.
  </Accordion>
  <Accordion title="Type and remote" icon="code-branch">
    A small row shows whether the project is **Local** or **GitHub**, alongside the Git remote when one is set. This tells you at a glance where the code lives.
  </Accordion>
  <Accordion title="Active state" icon="circle-check">
    The active project's card is highlighted, and its action reads **Active project** instead of **Make active**. Use this to confirm which project your team is currently working in.
  </Accordion>
</AccordionGroup>

## Track progress

You track a project by watching the work that lives inside it move forward.

<Steps>
  <Step title="Make the project active">
    Set the project as active so new work and chat are scoped to it.
  </Step>
  <Step title="Open Issues">
    Go to [Issues](/en/issues) to see every unit of work for the project and its current status, from backlog to done.
  </Step>
  <Step title="Group big work into epics">
    For larger goals, use [Epics](/en/epics) to bundle related issues so you can follow the whole effort as one thread.
  </Step>
  <Step title="Let the team execute">
    The orchestrator plans, specialists refine, and the local Forge applies code changes. Watch issues move through their statuses as the work lands.
  </Step>
</Steps>

<Info>
Progress is the sum of your issues and epics, not a separate number you maintain by hand. Keep work flowing as issues, and the project's status follows.
</Info>

## Remove a project

When a codebase is no longer in play, you can remove its project.

<Steps>
  <Step title="Hover the card">
    On the **Projects** page, hover the project card. A trash icon appears in the bottom right corner of the card.
  </Step>
  <Step title="Click the trash icon">
    Click it to start removal. Orkestral asks you to confirm, naming the project so you do not remove the wrong one.
  </Step>
  <Step title="Confirm">
    Confirm the prompt. The project is removed from the workspace. If it was the active project, the active selection is cleared.
  </Step>
</Steps>

<Warning>
Removing a project removes it from your workspace. It does not delete your actual code on disk or your remote repository; those stay exactly where they are.
</Warning>
