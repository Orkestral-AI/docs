The activity log is your audit trail. It answers the question every team eventually asks: what happened, who did it, and when. Orkestral records the work your agents do (chat replies, issue runs, heartbeats, code reviews, Forge executions) and surfaces it in two complementary views: a clean **Activity** timeline and a live **Logs** terminal with filters and metrics.

<Info>
Everything here is read from your local SQLite database in `~/.orkestral`. Nothing is sent anywhere. The activity log is a window into data you already own.
</Info>

## Two views, two jobs

Orkestral splits "what happened" into two screens because the questions are different. Pick the one that matches what you are trying to learn.

<CardGroup cols={2}>
  <Card title="Activity" icon="clock">
    A calm, human readable timeline of meaningful events: issues created, agents paused or resumed, skills attached, sessions started, heartbeats. Best for a high level "what changed today" recap.
  </Card>
  <Card title="Logs" icon="square-terminal">
    A live, technical stream with source filters, search, economy metrics, a per agent dashboard, and detailed Forge run cards. Best for auditing a specific run or watching work in real time.
  </Card>
</CardGroup>

## The Activity timeline

Open **Activity** to see the most recent events for your active workspace, newest first. The list refreshes on its own every 20 seconds, so you can leave it open and watch it fill in.

{/* PRINT: the Activity page showing a vertical timeline of recent events */}

Each row carries an icon, a human readable title, and a small metadata line underneath:

<Steps>
  <Step title="Icon and color">
    The icon tells you the kind of event at a glance. Green for an issue created, blue for a status change, yellow when an agent is paused, purple for a heartbeat run, red when a heartbeat fails.
  </Step>
  <Step title="Title">
    A short sentence describing the event, for example the issue that was created or the agent that changed state.
  </Step>
  <Step title="Metadata line">
    Three small facts: the **actor** (who or what triggered it), the raw event **kind** (shown in monospace, like `issue.created`), and the **timestamp**.
  </Step>
</Steps>

### Event kinds you will see

These are the kinds the timeline recognizes and gives a dedicated icon. Anything unrecognized still appears, with a neutral fallback icon.

<AccordionGroup>
  <Accordion title="Issues" icon="list-check">
    `issue.created` when a new issue lands, and `issue.status_changed` when an issue moves between states (for example backlog to in progress).
  </Accordion>
  <Accordion title="Agents" icon="robot">
    `agent.created` when you hire an agent, `agent.paused` when one is paused, and `agent.resumed` when it is brought back.
  </Accordion>
  <Accordion title="Skills" icon="puzzle-piece">
    `skill.attached` and `skill.detached` when you change which skills an agent carries.
  </Accordion>
  <Accordion title="Sessions" icon="comments">
    `session.created` when a new chat session is opened with an agent.
  </Accordion>
  <Accordion title="Heartbeats" icon="heart">
    `heartbeat.run` for a scheduled or manual heartbeat, and `heartbeat.failed` when one does not complete.
  </Accordion>
</AccordionGroup>

<Note>
The Activity timeline always scopes to your **active workspace**. If you switch workspaces, you see that workspace's events. With no active workspace selected, the page prompts you to pick one.
</Note>

## The Logs terminal

Open **Logs** for the deeper, live view. A small pulsing green dot in the header labeled **Live** tells you the stream is connected. New lines stream in as work happens, and the terminal auto scrolls while you are at the bottom. Scroll up to read history and a **New lines** button appears so you can jump back down when you are ready.

{/* PRINT: the Logs page with the economy strip, agent dashboard, and live terminal */}

The Logs page stacks several panels from top to bottom.

### Economy strip

When the local Forge has done work, a compact strip summarizes how much premium model cost it saved:

<CardGroup cols={2}>
  <Card title="Local resolved" icon="gauge">
    How many executions the bundled local model handled without calling a premium API.
  </Card>
  <Card title="Premium avoided" icon="bolt">
    Premium calls that were skipped because the local model succeeded.
  </Card>
  <Card title="Tokens saved" icon="chart-line">
    Estimated input plus output tokens you did not spend.
  </Card>
  <Card title="Estimated savings" icon="circle-check">
    A rough dollar figure, shown with a `≈` sign because it is an estimate against a reference premium price.
  </Card>
</CardGroup>

The strip also shows how many runs **escalated** to a premium model and a local success rate hint when available.

<Tip>
The dollar figure is a "give you a sense" estimate, not an invoice. It multiplies tokens avoided by a reference premium price, so treat it as a rough indicator of value, not exact accounting.
</Tip>

### Diagnostics

When there are issue runs, a diagnostics line shows aggregate health: total runs, the percentage resolved locally, and how many failed. Below it, heuristic findings flag runs that look stuck, suspicious, or failed, color coded by severity (red high, yellow medium, faint low). If everything looks healthy, it says so. This panel refreshes every 15 seconds and hides itself when there are no runs.

### Agent dashboard

A row of agent chips shows who is on the team and what each one is doing right now. The status dot reflects state: an agent error shows red, paused shows yellow, busy pulses in the agent's color, and idle is muted. A header counter tells you how many agents are **working now** or that all are idle. "Busy" is derived from the live stream: an issue run starting marks the agent busy, and a success or error marks it idle again.

### The live stream

The bottom panel is the terminal itself. Each line is dense but readable:

<Steps>
  <Step title="Timestamp">
    Precise to the millisecond, so you can correlate events tightly.
  </Step>
  <Step title="Source chip">
    A colored, uppercase tag for the event source (see the table below).
  </Step>
  <Step title="Context">
    The agent name and, when relevant, the issue number (for example `#42`).
  </Step>
  <Step title="Message">
    The event text, optionally prefixed with a scope. The text color encodes the level: red error, yellow warn, green success, neutral info, faint debug.
  </Step>
  <Step title="Duration">
    When a step is timed, its duration in milliseconds appears at the end of the line.
  </Step>
</Steps>

## Filtering and searching the logs

The terminal toolbar is where auditing gets fast. Combine a source filter with free text search to zero in on exactly what you need.

### Filter by source

Click a source chip to show only that kind of event. The default is **All**.

| Filter | Source | What it covers |
| --- | --- | --- |
| All | every source | the unfiltered stream |
| Forge | `forge` | local model executions (see the special view below) |
| Routing | `model-routing` | decisions about which model handled a task |
| Learning | `learning` | knowledge base and learning activity |
| Embedding | `embedding` | embedding and semantic search work |
| Issues | `issue` | issue run lifecycle events |
| Chat | `chat` | chat replies |
| Review | `review` | code review activity |

There is also a `system` source for internal events that appears in the **All** view.

### Search the text

Outside the Forge view, a search box filters the visible lines as you type. It matches against the message, the agent name, and the scope, so you can search for a phrase, an agent, or a sub step. A counter next to the box shows how many lines are visible out of the total currently buffered (the terminal keeps up to the most recent 1500 entries in memory).

<Warning>
Search and the line counter operate on what is loaded in the terminal, not your entire history. The stream caps at the latest 1500 entries, so very old events scroll out of the live buffer.
</Warning>

### The Forge view is special

Select the **Forge** filter and the raw log lines are replaced by readable run cards, one per local execution. This is the best way to audit code changes the local model made.

<AccordionGroup>
  <Accordion title="Local or premium badge" icon="sitemap">
    Each card is tagged as resolved locally or escalated to a premium model, so you can see at a glance where the work happened.
  </Accordion>
  <Accordion title="Validation result" icon="circle-check">
    A badge shows whether validation passed, failed, or was skipped after the change was applied.
  </Accordion>
  <Accordion title="Files and diff summary" icon="code-branch">
    Expand a card to see the files touched, a summary of the diff, and (when it failed) the reason the run escalated.
  </Accordion>
  <Accordion title="Run details" icon="gear">
    The expanded card also lists the execution mode, the risk level, the number of attempts, and the tokens avoided.
  </Accordion>
</AccordionGroup>

<Note>
The search box, line counter, and clear button are hidden in the Forge view, because that view is a curated list of runs rather than a raw text stream.
</Note>

## Clearing the trace

In the standard log view, a trash icon in the toolbar clears the visible trace and empties the buffer. Use it to start a clean recording before reproducing a problem, so the next lines are the ones that matter. Clearing affects the live buffer you are watching.

## How activity is assembled

Behind the scenes, an agent's activity is a merge of several data sources, normalized into one timeline. This is why a single agent's record can include very different kinds of work in one place.

<CardGroup cols={2}>
  <Card title="Chat runs" icon="comments">
    Messages the agent answered, linked back to their session.
  </Card>
  <Card title="Heartbeat runs" icon="heart">
    Scheduled or manual heartbeats, marked as such.
  </Card>
  <Card title="Code reviews" icon="magnifying-glass">
    Reviews where the agent was the reviewer, linked to the repo and PR.
  </Card>
  <Card title="Issue runs" icon="list-check">
    Executions where the agent (Forge or premium) actually did the work on an issue.
  </Card>
</CardGroup>

Each source uses its own status words, so Orkestral maps them into a shared vocabulary: **running**, **done**, **error**, **cancelled**, and **queued**. That normalization is what lets the timeline, the success rate, and the per kind counts treat every source consistently. Durations are computed from start and finish times, and items are sorted newest first before being returned.

<Info>
Status mapping in plain terms: a review marked `completed` and a heartbeat marked `succeeded` both become **done**, while a `failed` run of any kind becomes **error**. This keeps "did it work?" answerable across every kind of work an agent does.
</Info>

## Use it to audit

The activity log shines when you need to reconstruct what happened. A few patterns:

<Steps>
  <Step title="Recap the day">
    Open **Activity** and skim newest first. The icons and titles give you a quick narrative of issues, agent state changes, and heartbeats without any noise.
  </Step>
  <Step title="Investigate a failure">
    Open **Logs**, check the diagnostics line for failed runs, then filter to the relevant source (or **Forge**) and read the red lines or the failure reason on the run card.
  </Step>
  <Step title="Verify a code change">
    Filter to **Forge**, find the run, and expand it to confirm which files changed, whether validation passed, and how many attempts it took.
  </Step>
  <Step title="Confirm cost savings">
    Read the economy strip to see how much work stayed local and roughly how much premium spend you avoided.
  </Step>
</Steps>
