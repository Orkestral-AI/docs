A visão **Custos** responde a uma pergunta: quanto do seu trabalho rodou localmente de graça e quanto teve que escalar para um modelo premium pago. O Orkestral é local-first, então a maioria das mudanças de código é executada pelo **Forge** embutido (Qwen2.5-Coder) a `$0` de custo de API. Modelos premium só rodam quando o planejamento precisa deles, ou quando uma edição local não pode ser aplicada com segurança. Esta página mostra o dinheiro real que ficou no seu bolso por causa dessa divisão.

<Note>
Todo valor em dólar aqui é **real**, não estimado. O gasto premium vem do `cost_usd` efetivo informado pelo stream do modelo a cada run. Nada nesta página é fixado no código.
</Note>

## Como o número de economia funciona

O destaque desta página é a seção de **economia**. Ela só aparece quando já houve pelo menos um run orquestrado no workspace.

<Steps>
  <Step title="O Orkestral planeja com um modelo premium">
    O orquestrador CEO e os especialistas leem seu repo e decidem o que mudar. O planejamento é onde os modelos premium justificam seu custo.
  </Step>
  <Step title="O Forge executa a mudança localmente">
    Em vez de pagar um modelo premium para escrever a edição, o Forge local emite o diff e um motor determinístico o aplica aos seus arquivos. Esse run custa `$0` em taxas de API e é contado como uma execução local.
  </Step>
  <Step title="Premium roda só quando precisa">
    Se uma edição local não pode ser aplicada com segurança (as âncoras não batem), o run escala para um modelo premium. O `cost_usd` real desse run é registrado como gasto premium.
  </Step>
  <Step title="O Orkestral calcula o que você economizou">
    Ele pega o número de runs que o Forge resolveu, multiplica pelo custo médio que um modelo premium cobraria pelo mesmo trabalho e mostra isso como dinheiro economizado. Também mostra o que você de fato gastou nos runs que escalaram.
  </Step>
</Steps>

### Lendo os números

<AccordionGroup>
  <Accordion title="Economizado (o grande número esverdeado)" icon="bullseye">
    Os dólares estimados que você **não** gastou porque o Forge cuidou do trabalho localmente. É a contagem de runs locais multiplicada pelo custo premium médio por run observado neste workspace. Se o Orkestral ainda não viu um run premium para ancorar essa média, ele não consegue precificar a economia e mostra uma mensagem mais simples (veja abaixo).
  </Accordion>
  <Accordion title="Runs locais" icon="bolt">
    O número total de runs que o Forge resolveu sem um modelo pago. Isso combina **execuções totalmente locais** e runs **assistidos localmente** (casos em que o motor local ainda carregou a edição). Cada um destes é um run que você teria pago de outra forma.
  </Accordion>
  <Accordion title="Custo premium médio" icon="chart-line">
    O custo real médio de um run premium neste workspace, derivado de valores `cost_usd` efetivos. Este é o preço por run usado para valorizar seus runs locais. Como é medido, reflete os modelos que você realmente usa, não um preço de tabela genérico.
  </Accordion>
  <Accordion title="Premium gasto" icon="circle-dollar">
    O total real que você gastou em runs que escalaram para um modelo premium. Este é dinheiro que de fato saiu da sua conta, somado a partir do custo informado de cada run.
  </Accordion>
</AccordionGroup>

<Tip>
Uma proporção saudável é muitos **runs locais** contra um pequeno **premium gasto**. Isso significa que o Forge está absorvendo a maior parte do trabalho de execução e os modelos premium ficam reservados ao planejamento e às poucas edições que realmente precisam deles.
</Tip>

### Quando ainda não há dados suficientes

Se houve runs mas o Orkestral ainda não observou um run premium precificado para montar uma média, ele não consegue colocar um valor em dólar na sua economia. Nesse caso a seção ainda aparece e informa quantos runs rodaram localmente, só que sem um valor em dólar. O número de economia é preenchido assim que um run premium dá um custo real por run para se ancorar.

## Fases locais evitadas

{/* PRINT: Seção de fases locais mostrando tokens evitados */}

Um segundo bloco de economia aparece quando o Forge rodou **fases locais**. Parte do trabalho de múltiplas etapas pode ser carregada inteiramente pelo modelo local em vez de passar por streaming por um modelo premium. Quando isso acontece, o Orkestral rastreia os tokens que essas fases teriam enviado a um modelo pago e os reporta como **tokens evitados**, mostrados em milhares (por exemplo `~42k tokens`), junto com o número de runs de fases locais.

<Info>
Tokens evitados é uma métrica de volume, não de dólar. Ela diz quanto tráfego de modelo premium o Forge deslocou. Quanto mais fases locais rodam, mais volume de tokens nunca chega a uma API paga.
</Info>

## Por que o Forge pode rodar de graça

Modelos premium sabem escrever código, mas pagar um para aplicar uma edição rotineira é desperdício. O motor **Fast Apply** do Orkestral deixa um pequeno modelo local fazer o mesmo trabalho de forma determinística.

<Tabs>
  <Tab title="A divisão">
    Modelos premium **planejam**: eles leem o contexto e decidem o que deve mudar. O Forge **executa**: ele emite o diff de fato. Como o Forge roda na sua máquina, a execução não custa nada em taxas de API. A página de Custos é o placar exatamente dessa divisão.
  </Tab>
  <Tab title="Aplicação determinística">
    O Forge emite uma edição "lazy": apenas as linhas que mudam, com algumas linhas reais ao redor mantidas como **âncoras** e o resto marcado como inalterado. Um motor de merge determinístico encontra essas âncoras no seu arquivo e substitui apenas o trecho entre elas. O app aplica a edição, nunca o modelo, então a mesma entrada sempre produz o mesmo arquivo.
  </Tab>
  <Tab title="Seguro por design">
    O merge casa âncoras em camadas: exata, depois normalizada por espaços em branco, depois uma correspondência aproximada fortemente limitada que só se aplica quando há exatamente um candidato confiável. Se nada casar de forma limpa, ele se recusa a adivinhar e o run escala para premium. É por isso que escalações são raras mas sempre corretas: o motor prefere pagar a escrever o código errado.
  </Tab>
</Tabs>

<Warning>
Existe um fallback opcional de fast-apply hospedado (Morph) que algumas configurações habilitam via uma variável de ambiente antes de escalar para premium. Ele vem desligado por padrão e nunca é ligado pela UI, porque o Orkestral é local-first. Você não precisa dele para a economia desta página funcionar.
</Warning>

## Como reduzir ainda mais seus custos

<CardGroup cols={2}>
  <Card title="Contrate especialistas, planeje uma vez" icon="users">
    Deixe o orquestrador planejar um lote de trabalho e então deixe o Forge executar cada passo. Uma rodada de planejamento premium pode disparar muitas edições locais gratuitas.
  </Card>
  <Card title="Mantenha edições ancoráveis" icon="code-branch">
    Arquivos limpos e bem estruturados dão âncoras estáveis ao Forge, então mais edições se aplicam localmente e menos escalam. Código espalhado ou duplicado torna as âncoras ambíguas e força runs premium.
  </Card>
  <Card title="Use heartbeats de forma deliberada" icon="clock">
    Agentes agendados fazem runs reais. Defina intervalos que combinem com a velocidade real de mudança do seu repo para não pagar por escalações premium em ciclos ociosos.
  </Card>
  <Card title="Acompanhe a proporção ao longo do tempo" icon="gauge">
    Confira esta página depois de uma sessão movimentada. Uma contagem local em alta contra um premium gasto estável é o sinal de que o Forge está cumprindo seu papel.
  </Card>
</CardGroup>

## O que vem a seguir

Mais detalhamentos de custo estão a caminho (a página reserva um espaço para eles). Por ora, os blocos de economia e de fases locais são a fonte da verdade de quanto o Forge manteve fora da sua conta.

Para passar mais trabalho pelo Forge, vá aos seus [agentes](/pt/agents) e [issues](/pt/issues), planeje com o orquestrador e volte aqui para ver a contagem local subir.
